Welcome to the Restaurant menu app

A small, deliberately modern React 19 + TypeScript single-page app that renders a restaurant menu you can search and filter by tag. It was built as a workshop exercise, so the code is tidy and heavily commented — but it uses a bleeding-edge stack and has a couple of footguns worth knowing before your first change.

01 The stack

Everything is current-generation and moves fast. When docs or Stack Overflow disagree with the code, trust the code — these versions are newer than most answers online.

UI

React 19.2

Rendered in StrictMode. No class components, no Redux — plain function components.

Language

TypeScript ~6.0

Strict, bundler module resolution, noEmit. Vite does the transpiling.

Build & dev server

Vite 8

HMR in dev; tsc -b && vite build for prod. Config in vite.config.ts.

Routing

TanStack Router 1.x

File-based routes in src/routes/, auto code-splitting, type-safe URL search params.

Validation

Zod 4

Validates the URL's ?q= and ?tags= search params — never throws, degrades gracefully.

Styling

Tailwind CSS 4

CSS-first config — no tailwind.config.js. Theme tokens live in src/index.css.

Images

vite-imagetools

Generates resized WebP + srcSet at build time from source JPGs, for Lighthouse performance.

Testing

Vitest 4 + Testing Library

jsdom environment. Config is folded into vite.config.ts, not a separate file.

Linting

Oxlint

Not ESLint. Rust-based, fast. Run npm run lint. Config in .oxlintrc.json.

Formatting

Prettier

Tabs, LF line endings (enforced via .gitattributes), with the Tailwind class-sorting plugin.

04 How it works

The one architectural idea to internalize: the URL is the source of truth for filter state — there is no useState for the search query or selected tags.

URL
?q & ?tags
→
Zod
search-schema
→
Route
useSearch()
→
Pure fn
filterFoods()
→
UI
MenuGrid

The index.tsx route validates the incoming search params with the Zod schema, reads them via Route.useSearch(), and passes them to the pure filterFoods(foods, {q, tags}). Changing a filter navigates (updates the URL) rather than setting local state — which is why searches are shareable and bookmarkable, and the back button works.

  • Typing in the search box uses replace (no history spam); toggling a tag pill uses push (a deliberate, back-able action).
  • Filtering is tags OR'd together, AND'd with a case-insensitive text match across name, description, and tags.
  • filterFoods and the Zod schema are plain functions with no React — that's what makes them cheap to unit-test.
  • Prices always go through formatPrice() (never .toFixed) and copy through t() — these are deliberate localization seams for a future i18n library.
  • Dark mode is driven purely by the OS prefers-color-scheme via CSS variables — no dark: classes and no toggle in the UI.

05 Red flags & easy-to-miss things

Read these before your first PR. Most are one-line traps that cost an afternoon.

Good to know

Adding a menu item is a two-file change

FoodCard looks up its image with foodImages[food.image]. If you add a dish to food.ts whose image filename isn't also imported in food-images.ts (which uses explicit per-image imports, not a glob), the lookup is undefined. FoodCard now guards this and renders a blank placeholder tile instead of crashing — but the dish stays image-less until you add the import, so still update both files together.

Don't touch

src/routeTree.gen.ts is generated

The TanStack Router Vite plugin regenerates it on every dev/build. Never edit it by hand — add routes by adding files under src/routes/. It's linter/formatter-ignored on purpose.

Config gotcha

Plugin order in vite.config.ts matters

tanstackRouter() must be listed before react() so it can transform route files first. Reordering them silently breaks routing. A comment in the file warns you — heed it.

No config file

Tailwind v4 has no tailwind.config.js

If you go looking for one, you'll waste time. All theme tokens (colors, fonts) are declared in the @theme block inside src/index.css, and dark-mode values are a prefers-color-scheme override in the same file.

Tooling surprise

It's Oxlint, not ESLint

There's no .eslintrc. Lint config is .oxlintrc.json and the command is npm run lint. Same idea, different (much faster) tool — don't try to bolt ESLint plugins onto it.

Stale docs

The README and specs lie a little

README.md is still the stock Vite template, and claude.md is empty — neither describes this app. The design spec in docs/superpowers/specs/ even says "no search or filtering, display only," which is no longer true (search/filter were added after). Treat this page as the current picture, not those.

Easy to miss

Tests colocated & kept out of the route tree

A *.test.tsx sits next to each source file — including inside src/routes/. Those test files are excluded from routing via routeFileIgnorePattern in the Vite config, so a test file won't accidentally become a route.

Good to know

Bad URLs won't crash the page

The Zod schema uses .catch() and strips empty values, so a rotted or hand-edited link (unknown tags, ?q= with no value) quietly falls back to defaults and still renders the full menu. Empty filters are removed from the URL to keep it clean.

First good task to get your bearings: add a new dish. You'll touch food.ts (data) and food-images.ts (image import) together, drop a JPG into src/assets/food-images/, and add a test. That single change exercises the data model, the image pipeline, and the biggest footgun above.
